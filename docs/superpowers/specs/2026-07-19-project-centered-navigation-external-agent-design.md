# 项目中心导航与外部智能体设计

## 1. 目标

将 KOL Campaign OS 从“按后台功能拆分的工具集合”调整为“以项目为中心的业务工作台”。用户不需要在多个一级页面之间手工串联产品、策略、寻找、候选和合作流程。

系统本身暂不提供智能体聊天界面。外部智能体通过 MCP 或 Skills 封装的 HTTP API 与系统连接，负责自然语言理解和策略推理；KOL Campaign OS 负责数据、业务规则、状态、审计和人工审批。

## 2. 设计原则

1. Campaign 是日常工作的主要入口，一个 Campaign 可以关联多款 Product。
2. 每个活跃的 Campaign Product 拥有独立的产品需求和 KOL 策略。
3. 外部智能体自动生成策略草案，用户确认后才能启动 KOL 寻找。
4. 多款产品分别运行寻找流程，但共享 KOL 主库、身份去重和历史合作信息。
5. 同一个 KOL 可以在同一 Campaign 中合作多款产品，联系与谈判记录保持一条，产品分配保持多条。
6. 内容分析是跨项目能力，不作为 KOL 寻找的附属模块。
7. 用户界面使用中文业务名称；数据库、API 和代码可继续保留 `brief`、`finder`、`kol` 等稳定字段名。

## 3. 一级导航

一级导航收敛为五项：

- 项目
- KOL 库
- 产品库
- 内容分析
- 设置

取消独立 Dashboard。登录后默认进入项目列表，项目列表同时承担首页职责，展示进行中的项目和需要人工处理的事项。

以下现有功能不再作为一级导航：

- KOL 策略
- KOL 寻找
- KOL 合作
- 视频数据
- 分析记录
- AI Prompt 模板
- API 设置

其中前三项进入项目内部；视频与分析合并为“内容分析”；Prompt、API 和 Agent 权限进入“设置”。

## 4. 项目内部导航

每个项目提供五个页面：

1. **项目概览**：项目状态、市场、时间、产品数量、策略状态、寻找进度、待审批事项和异常。
2. **产品需求**：关联产品、主推/辅推角色、状态、优先级，以及每款产品独立的推广目标、核心卖点、目标受众、内容要求和限制。
3. **KOL 策略**：按 Campaign Product 展示外部智能体生成的策略草案、版本、校验结果和确认状态。
4. **KOL 寻找**：展示各产品的寻找任务、视频证据、匹配分、候选列表和候选审批。
5. **KOL 合作**：管理 Campaign 级联系与谈判记录，以及该 KOL 在不同 Campaign Product 下的分配和合作状态。

页面步骤可概括为：

`产品需求 → KOL 策略 → KOL 寻找 → 候选审批 → KOL 合作`

“候选审批”是 KOL 寻找页面内的阶段，不单独出现在项目导航中。

## 5. 外部智能体架构

### 5.1 责任边界

外部智能体负责：

- 与用户对话并理解推广需求；
- 补问缺失信息；
- 查询已有项目和产品；
- 判断复用产品或建议新建产品；
- 为每款活跃产品生成独立的结构化 KOL 策略草案；
- 根据用户反馈修改草案；
- 在用户确认后调用创建、发布或执行接口；
- 查询进度并向用户汇报。

KOL Campaign OS 负责：

- 保存 Campaign、Product、Campaign Product、Strategy、Finder Task、KOL 和合作数据；
- 校验关联关系、字段、枚举、状态和策略版本；
- 防止重复创建、重复关联和重复执行；
- 执行寻找任务、证据分析、身份去重和匹配归因；
- 保存操作来源和审计信息；
- 强制执行人工确认与审批边界。

### 5.2 单一业务能力层

HTTP API 是唯一业务能力层，MCP 与 Skills 只封装这些 API，不复制业务规则：

`用户 ↔ 外部智能体 ↔ MCP / Skill ↔ HTTP API ↔ 数据库与任务执行器`

系统暂不保存完整聊天记录，也不实现聊天 UI。

### 5.3 策略来源审计

外部智能体写入的策略至少记录：

- `source_type: external_agent`
- `source_agent`
- `source_request_id`
- `created_by`
- 策略版本
- 创建与确认时间

所有写请求使用幂等键，防止智能体重试造成重复项目、重复策略或重复寻找任务。

## 6. 对话式创建项目

项目可以由外部智能体通过自然语言对话创建，系统网页仍可保留传统表单作为备用入口。

标准流程：

1. 用户向外部智能体描述品牌、市场、时间、目标平台和产品。
2. 智能体提取结构化项目资料，并查询系统内是否已有相同 Campaign 或 Product。
3. 智能体向用户展示确认摘要，包括将创建的项目、复用或新建的产品、产品状态、市场、时间和推断字段。
4. 用户确认后，智能体调用事务式 API 创建 Campaign、创建或复用 Product，并建立 Campaign Product 关联。
5. 智能体为每款活跃 Campaign Product 分别生成 KOL Strategy Draft，并写回系统。
6. 系统校验草案并返回缺失资料、风险和字段错误。

如果某款产品资料不足，该策略状态为“需要补充资料”；它不能阻塞同一项目中其他产品生成草案。

## 7. 策略确认与 KOL 寻找

策略采用“两阶段”执行：

### 7.1 草案阶段

- 外部智能体可以创建和更新策略草案。
- 系统只校验和保存，不启动寻找任务。
- 用户可以通过外部对话或网页修改草案。
- 每次修改生成新版本或更新版本号。

### 7.2 确认阶段

- 用户确认具体策略版本后，外部智能体调用发布接口。
- 如果确认期间策略已被修改，系统拒绝发布旧版本。
- 策略发布成功后，系统才允许创建对应 Campaign Product 的寻找任务。
- 同一产品、平台和有效策略版本不得重复启动等价任务。

AI 不得自动审批候选 KOL。候选审批、建立正式合作和归档合作仍由人工完成。

## 8. KOL 复用与多产品合作

寻找以单款 Campaign Product 为执行单位，但入库和去重使用共享 KOL 主库。

发现候选时，系统应标记：

- 新 KOL；
- 已有 KOL，首次匹配当前产品；
- 已合作 KOL，再次匹配；
- 已在同项目其他产品中出现。

审批后：

- Campaign 中只维护一条 Campaign KOL 联系与谈判记录；
- 通过 Campaign KOL Product 为该 KOL 关联一款或多款产品；
- 每款产品独立保存匹配分、证据、分配状态和合作结果。

## 9. 内容分析

“视频数据”和“分析记录”合并为独立一级模块“内容分析”。它可以支持：

- KOL 寻找的视频证据分析；
- 竞品分析；
- 品类洞察；
- 卖点研究；
- KOL 内容分析；
- 合作内容复盘。

用户向外部智能体直接输入研究目标。智能体自动判断分析类型，同时允许用户修改。执行前应确认：

- 分析类型；
- 研究对象；
- 平台与市场；
- 时间范围；
- 是否关联项目、产品或 KOL；
- 建议搜索词；
- 预计分析的视频数量。

内容分析内部包含：

- 新建分析；
- 分析任务；
- 视频库；
- 分析结果。

除 KOL 寻找证据外，内容分析不强制关联 Campaign、Product 或 KOL。项目页面只展示与当前项目相关的分析摘要和跳转入口。

## 10. 设置

设置模块统一承载：

- API 与平台连接；
- Prompt 与结构化输出规范；
- MCP / Agent 身份和权限；
- 系统配置。

外部智能体权限分级：

- 可自动执行：查询、资料完整度检查、策略草案写入、进度读取；
- 用户确认后执行：创建项目、创建产品、关联产品、发布策略、启动寻找；
- 仅人工审批：通过 KOL、建立正式合作、归档合作；
- 禁止执行：删除用户业务数据、绕过审批、覆盖人工结论。

## 11. 错误与并发处理

- 事务式创建项目失败时，不得留下半创建的 Campaign Product 关系。
- 产品重复或身份不确定时，返回候选匹配，不自动创建近似重复产品。
- 策略发布使用版本校验，避免覆盖人工或其他智能体的新修改。
- 寻找任务失败只影响对应 Campaign Product，不影响项目内其他产品。
- 所有 Agent 写入记录来源、请求标识和错误原因，支持审计与安全重试。

## 12. 验收标准

1. 一级导航只显示项目、KOL 库、产品库、内容分析和设置。
2. 登录默认进入项目列表，不再提供独立 Dashboard。
3. 用户可以在一个项目内完成产品需求、KOL 策略、KOL 寻找和 KOL 合作管理。
4. 外部智能体可以通过统一 HTTP API 创建项目及关联产品，并写入每款产品的策略草案。
5. 未确认的策略不能启动寻找任务；过期策略版本不能发布。
6. 同一个 KOL 可以匹配和合作多款产品，同时只保留一条 Campaign 级联系记录。
7. 内容分析可以脱离项目独立运行，也可以选择关联项目、产品或 KOL。
8. 所有候选审批和正式合作决策仍由人工完成。
