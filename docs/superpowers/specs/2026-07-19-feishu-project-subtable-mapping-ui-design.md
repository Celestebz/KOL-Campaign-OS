# 飞书项目子表映射 UI 设计

## 背景

API 设置页当前用“默认项目 KOL 子表”输入框和“项目子表映射”文本框配置飞书数据表。后端已经支持按项目 ID 或项目名称选择子表，但手写 `项目名称=tbl...` 容易拼错，也让用户误以为系统只能配置一张项目子表。默认子表还会让未配置的项目静默写入同一张表，增加数据混写风险。

## 目标

- 将手写文本映射改为可增删的紧凑行编辑器。
- 项目只能从系统现有项目中选择，不能手动输入。
- 每个项目绑定一张飞书 KOL 子表。
- 移除默认子表的界面和同步兜底行为。
- 未配置映射的项目同步时返回明确错误。
- 兼容读取现有名称映射、项目 ID 映射和行文本格式，不要求数据库迁移。
- 不删除现有业务数据或已保存的飞书凭证。

## 非目标

- 不创建或删除飞书多维表格。
- 不自动修改飞书字段结构。
- 不新增数据库表或专用映射 API。
- 不实现飞书连接测试、字段远程校验或自动发现 Table ID。
- 不修改 KOL 总表和项目表的同步字段。

## 方案选择

采用“前端结构化编辑、继续保存现有映射字段”的方案。

设置页从 `/api/campaigns` 获取项目列表，在前端把 `campaign_subtable_map` 解析成编辑行。保存时仍写回现有配置字段，但统一序列化为以项目 ID 为键的 JSON：

```json
{
  "1": "tbliXAzgY46zjt3U",
  "2": "tblAAAAAAA"
}
```

该方案利用后端现有的项目 ID 查找能力，避免数据库迁移，同时让项目改名不再破坏新保存的映射。

## 用户界面

### 布局

云端存储区域移除“默认项目 KOL 子表”输入框。保留 App ID、App Secret、OpenAPI Base URL、Base/App Token、KOL 总表 ID 和项目表 ID。

“项目子表映射”改为紧凑行编辑器。每行包含：

1. 系统项目下拉框；
2. 飞书子表 ID 输入框；
3. 删除按钮。

列表底部提供“添加项目映射”按钮。没有映射时显示简短空状态，说明每个需要同步的项目都必须配置独立子表。

### 项目选择

- 下拉选项来自 `/api/campaigns`。
- 不允许自由输入项目名。
- 已被其他行选择的项目在下拉中禁用，防止重复映射。
- 项目列表加载失败时显示警告并禁用新增映射；飞书其他配置仍可查看和保存。

### 校验

保存云端存储设置前执行以下校验：

- 每行必须选择系统项目；
- 每行必须填写 Table ID；
- Table ID 去除首尾空格后必须以 `tbl` 开头；
- 同一项目不能出现两次。

错误显示在对应控件附近，保存被阻止，已填写内容保留。

## 数据兼容

### 读取

前端解析器兼容：

- 以项目 ID 为键的 JSON；
- 以项目名称为键的 JSON；
- 每行一个 `项目名称=tbl...` 的旧文本格式。

名称键通过当前系统项目列表解析为项目 ID。无法解析的旧映射不静默丢弃：设置页显示警告，列出无法匹配的名称；用户处理前不自动覆盖这些条目。

### 保存

保存时将有效编辑行序列化成项目 ID JSON，继续写入 `cloudStorage.feishu.campaign_subtable_map`。现有设置接口和数据库字段保持不变。

`campaign_kol_table_id` 不在新界面展示，也不参与同步兜底。原值可以继续保存在配置中，但运行时忽略，避免因为 UI 改版而主动删除已有设置数据。

## 同步行为

项目 KOL 同步只按以下顺序查找目标表：

1. `campaign_id` 数字键；
2. `campaign_id` 字符串键；
3. 兼容期内的项目名称键。

不再回退到 `campaign_kol_table_id`。没有映射时，该条同步失败并返回：

```text
项目“<项目名称>”尚未配置飞书 KOL 子表
```

本地 KOL 和项目数据保持不变，同步状态沿用现有失败处理逻辑。

## 组件边界

新增独立的项目子表映射编辑组件，职责为：

- 渲染映射行；
- 管理新增、删除和项目去重；
- 把表单值保持为结构化数组；
- 展示行级校验错误。

映射解析和序列化使用独立纯函数，避免把兼容逻辑堆入设置页组件，也便于单元测试。设置页负责加载项目列表、把远端配置转换为编辑值，以及保存前序列化。

## 错误处理

- 项目列表加载失败：显示非阻塞警告，禁用映射编辑，不清空已加载设置。
- 旧名称无法匹配：显示明确警告并阻止覆盖映射，直到用户处理。
- 表单校验失败：定位到对应行，不发起保存请求。
- 保存接口失败：沿用现有错误提示，保留当前编辑内容。
- 同步缺少映射：只将相关同步项标记为失败，不影响本地数据或其他已正确映射的项目。

## 测试

前端测试覆盖：

- 加载系统项目并渲染现有映射；
- 旧名称文本和名称 JSON 能解析为项目 ID；
- 添加、删除映射行；
- 已选项目不能重复选择；
- 空项目、空 Table ID 和非法前缀阻止保存；
- 保存请求包含以项目 ID 为键的 JSON；
- 未解析旧映射时不覆盖原配置；
- 项目列表加载失败时显示警告。

后端测试覆盖：

- 项目 ID 映射优先命中；
- 旧项目名称映射仍可命中；
- 缺少映射时不再使用默认子表；
- 缺少映射时返回包含项目名称的明确错误。

最后运行设置页相关测试、同步路由测试和前端生产构建。

## 验收标准

- 用户可为任意数量的现有系统项目配置独立的飞书子表。
- 项目名称不能手输，重复项目不能保存。
- 新保存的映射使用项目 ID，项目改名不影响同步。
- 未配置的项目不会写入默认表，而是明确同步失败。
- 旧映射可兼容读取，未识别条目不会被静默丢失。
- KOL 总表、飞书凭证及本地业务数据不受影响。
