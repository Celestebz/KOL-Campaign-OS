# API 设置页信息架构与交互改版设计

日期：2026-07-17

## 背景

当前 API 设置页把平台数据源、AI 模型、Agent、External Agent、云端存储和备用策略的全部字段连续铺开。Provider 的启用状态、配置状态和预留状态处于同一视觉层级，导致页面过长、查找困难，也难以判断哪些配置真正参与当前运行。

本次改版只调整前端信息架构和配置交互，不改变后端设置接口、字段路径、默认值、密钥更新规则或现有用户数据。

## 目标

- 用户无需长距离滚动即可定位任意设置类别。
- 用户能快速识别当前启用、已配置、待配置和预留的 Provider。
- 每次只编辑一个明确的配置对象，降低误改风险。
- 保留现有 `/api/settings` 保存语义，避免设置数据迁移。
- 桌面端和窄屏环境都能稳定使用。

## 非目标

- 不新增、删除或替换 Provider。
- 不修改 API Key、Token 或 Secret 的后端存储逻辑。
- 不改变 Finder、视频分析、飞书同步或 Fallback 的业务行为。
- 不实现新的 Provider 健康检查接口；概览状态只基于现有配置数据计算。

## 总体方案

采用“左侧分区导航 + 右侧内容工作区 + Provider 配置抽屉”。设置页不使用传统页码翻页，因为设置项不是线性流程，页码无法表达类别含义。

桌面端左侧显示稳定的分区导航，右侧一次只渲染一个分区。窄屏下左侧导航替换为顶部下拉选择，配置抽屉占满可用宽度。

## 信息架构

设置页划分为七个分区：

1. **概览**：配置健康度、当前启用项、缺失配置和快捷入口。
2. **平台数据源**：YouTube、Instagram、TikTok 的主数据源、备用数据源和 Provider 配置。
3. **AI 模型**：当前模型和 AI Provider 配置。
4. **Agent 自动化**：默认 Agent 和自动化 Provider 配置。
5. **External Agent API**：启用状态、API Token、接口用途说明和备注。
6. **云端存储**：飞书应用、Base、数据表映射和同步备注。
7. **运行与备用策略**：Fallback、失败原因、原始响应和 AI 工具调用等运行选项。

“备用策略”更名为“运行与备用策略”，使其同时涵盖 Fallback 和运行记录类选项。

## 概览

概览使用紧凑状态块展示：

- 平台数据源：分别展示三个平台当前主数据源及配置状态。
- AI 模型：展示当前启用模型及配置状态。
- Agent：展示默认 Agent 及配置状态。
- External Agent API：展示启用状态和 Token 是否已配置。
- 云端存储：展示飞书核心字段是否齐全。
- 运行与备用策略：展示 Fallback 是否启用。

每个状态块提供进入对应分区的快捷入口。状态只表达现有配置的完整度，不声称外部服务一定可用。

## Provider 卡片

平台数据源、AI 模型和 Agent 自动化统一使用 Provider 卡片。卡片只显示：

- Provider 名称与用途。
- 是否为当前主数据源、当前模型或默认 Agent。
- 已配置、待配置或预留状态。
- 一句关键配置摘要，例如“API Key 与 Connection ID 已配置”。
- “配置”操作。

状态优先级为：当前启用 > 配置完整度 > 预留。当前启用卡片使用明确边框和状态标签，不依赖颜色单独传达状态。

预留 Provider 默认隐藏。用户可通过“显示预留项”开关查看。若某个预留 Provider 已存在历史配置，即使开关关闭也必须显示，避免用户无法找到已保存数据。

## 平台数据源分区

平台数据源分区顶部使用 YouTube、Instagram、TikTok 三个语义明确的切换项，不使用页码。

每个平台工作区包含：

- 当前主数据源选择。
- Fallback 状态摘要。
- 已接入和已配置的 Provider 卡片。
- 备用数据源选择。
- “显示预留项”开关。

切换平台不会丢失当前尚未保存的表单状态。

## 配置抽屉

点击 Provider 卡片的“配置”后，从右侧打开抽屉。抽屉按 Provider 类型复用现有字段：

- 通用：API Key、Base URL。
- AI Provider：Model。
- Maton Gateway：Connection ID。
- Custom Provider：名称、认证 Header、认证 Scheme、备注。

抽屉标题同时显示 Provider 名称和所属范围，例如“YouTube 数据 Provider”。密钥字段不回显真实值，并明确说明“留空保留现有密钥；输入新值才更新”。

抽屉提供“取消”和“保存此配置”。保存期间按钮进入 loading 状态并禁止重复提交。保存失败时抽屉保持打开，在抽屉内展示错误信息；成功后关闭抽屉、重新获取设置并刷新卡片状态。

## 保存与数据流

后端目前通过 `POST /api/settings` 保存完整 Settings。本次不改变该接口。

保存流程：

1. 页面通过 `GET /api/settings` 加载完整配置，并与前端默认值合并。
2. 用户只在当前分区或抽屉编辑相关字段。
3. 点击局部保存时，前端把当前变更合并进完整 Settings 对象。
4. 前端使用现有 `{ settings: ... }` 结构调用 `POST /api/settings`。
5. 成功后重新获取服务端配置，避免本地状态与密钥占位规则不一致。

主数据源、当前 AI 模型、默认 Agent、External Agent 开关和运行策略等跨字段设置在所属分区内提供就近保存按钮，不保留依赖长滚动的全页底部保存按钮。

页面存在未保存修改时，切换分区允许保留内存状态；关闭或离开页面时显示未保存更改确认。取消抽屉只撤销该抽屉自打开后的临时修改，不影响此前已经保存的配置。

## 组件边界

- `Settings`：完整配置加载、默认值合并、未保存状态、分区路由和统一保存方法。
- `SettingsSectionNav`：桌面分区导航与窄屏下拉选择。
- `SettingsOverview`：只读配置完整度和快捷入口。
- `ProviderCard`：Provider 状态摘要与配置入口。
- `ProviderDrawer`：按 Provider 能力渲染字段并提交局部修改。
- `PlatformProviderSection`：平台切换、主源、备用源及 Provider 列表。
- `AiProviderSection`：当前模型及 AI Provider 列表。
- `AgentProviderSection`：默认 Agent 及 Agent Provider 列表。
- 其余分区组件：External Agent、云端存储、运行与备用策略。

Provider 元数据从当前散落的 Options 和 Label 定义整理为统一配置结构，用于生成选择项、卡片和抽屉字段；不改变发送到后端的字段路径。

## 响应式与可访问性

- 桌面端使用固定宽度的分区导航和弹性内容区。
- 窄屏下分区导航改为顶部 Select，Provider 卡片单列显示，抽屉占满宽度。
- 所有状态包含文字，不只依赖颜色。
- 导航、卡片操作和抽屉按钮提供键盘焦点样式。
- 抽屉关闭后焦点返回触发它的“配置”按钮。

## 错误与边界状态

- 初次加载失败：在工作区显示可重试错误状态，不展示空白表单。
- 保存失败：保留用户输入并显示服务端错误。
- Provider 未配置：卡片显示“待配置”，不伪装为服务不可用。
- Provider 已有部分字段：显示“配置不完整”，并在抽屉标出缺失字段。
- 预留 Provider 已有历史配置：始终可见。
- 未保存修改：离开页面前确认。

## 测试与验收

前端测试至少覆盖：

- 七个分区的切换与一次只显示一个分区。
- 平台切换及状态不丢失。
- 预留项默认隐藏、手动显示、历史配置强制显示。
- Provider 卡片的启用、完整、缺失和预留状态。
- 抽屉按 Provider 类型显示正确字段。
- 密钥字段不回显真实值，留空保存保留原语义。
- 局部保存正确合并完整 Settings 并使用现有请求结构。
- 保存成功刷新状态，保存失败保留抽屉和输入。
- 未保存修改的离开确认。
- 窄屏导航和抽屉布局。

最终运行相关前端测试和生产构建。验收时确认现有 Settings 数据无需迁移，Finder、分析和同步模块读取到的字段路径保持不变。

## 实施范围

主要修改 `client/src/pages/Settings.js`，并按现有项目组织方式增加必要的 Settings 子组件、样式和测试。只进行服务于本次改版的局部拆分，不做无关页面或后端重构。
